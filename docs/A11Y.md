# Accessibility (WCAG 2.1 AA) checking

Accessibility is a hard constraint for this project: every result renders as headline
text + an accessible `<table>` alongside the charts, the layout carries a skip link and
landmark structure, and forms wire `aria-invalid` / `aria-describedby`. This doc covers
how to check it.

## The authoritative check: axe DevTools / Lighthouse in your browser

For the go-live verification pass, run **current axe in the Chrome you already have** — it
is trustworthy, needs no downloads, and covers the authenticated forecast pages trivially
(just log in first):

- **axe DevTools** (free Deque browser extension): open a page, DevTools → *axe DevTools* →
  *Scan all of my page*. Do this for the landing + auth screens, then log in (the seeded
  `demo@example.com` / `password`) and scan `/dashboard`, `/scenarios/create`, a results
  page, `/account/security`.
- **Lighthouse** (built into Chrome DevTools): DevTools → *Lighthouse* → tick *Accessibility*
  → *Analyze*.

Fix the markup the report points at, not the report.

## Automated sweep (Pa11y CI)

[`.pa11yci.json`](../.pa11yci.json) drives [Pa11y CI](https://github.com/pa11y/pa11y-ci)
(axe runner, WCAG2AA) over the public pages + `/welcome`, with `npm run a11y`, and
[`.github/workflows/a11y.yml`](../.github/workflows/a11y.yml) runs it on push/PR.

**The headless browser must be installed explicitly.** `.npmrc` sets `ignore-scripts=true`
(a deliberate supply-chain hardening), so `npm ci` / `npm install` never runs puppeteer's
postinstall and **no browser is downloaded — locally or in CI** (an earlier note here wrongly
assumed CI would download one; it does not). The workflow installs one with `npx puppeteer
browsers install chrome` after `npm ci`, and puppeteer finds it in its cache automatically. To
run the sweep locally, either do the same, or point Pa11y at your installed Chrome:
```powershell
$env:PUPPETEER_EXECUTABLE_PATH = "C:\Program Files\Google\Chrome\Application\chrome.exe"
php artisan db:seed --class="Database\Seeders\DemoScenarioSeeder"   # demo@example.com / password
php artisan serve            # in one shell
npm run a11y                 # in another
```

Pa11y CI is a coarse regression smoke; **axe DevTools (current axe) in the browser remains the
authoritative check** — it covers the authenticated forecast pages and the chart canvases Pa11y
does not. The `/welcome` URL logs in as the demo user and needs that user to **not** yet have
accepted the disclaimer (true on a fresh `migrate:fresh --seed`, as in CI; a demo user who has
already accepted it makes that one URL time out locally).

## Sweep log

- **2026-06-29 (CI activated + fixed):** the repo gained a GitHub remote and the a11y workflow
  ran for the first time, and failed for two reasons: (1) `ignore-scripts` meant no browser was
  installed for Pa11y (the assumption that CI would download one was wrong), and (2) the bundled
  axe-4.2 would have flagged the public-page color-contrast false positive anyway. Fixed by
  bumping **pa11y-ci 3.1.0 → 4.1.1** (current puppeteer + current axe-core: the false positive
  is gone, all public pages now pass 0 errors) and adding an explicit `npx puppeteer browsers
  install chrome` step. Validated locally against installed Chrome: **5/5 URLs pass** with a
  fresh-seed (unacknowledged) demo user.
- **2026-06-29:** go-live browser pass (axe DevTools). One **serious** finding: the ten
  horizontally-scrollable data-table wrappers (`overflow-x-auto`) across the results and
  compare views were not keyboard-focusable (axe `scrollable-region-focusable`, WCAG 2.1.1 /
  2.1.3) — a keyboard-only user could not scroll an overflowing table. Fixed by adding
  `tabindex="0"` to each wrapper; the tables already carry `sr-only` captions inside labelled
  `<section>`s, so no nested region landmark was needed. Pa11y CI smoke alongside it surfaced
  only the known axe-4.2 color-contrast false positive on the public pages, plus a `/welcome`
  timeout that is the scripted-login step landing on an **already-acknowledged** demo user
  (the disclaimer screen redirects away, so the awaited form button never appears) — a Pa11y
  fixture limitation, not an app fault.
- **2026-06-28:** first local sweep. Found and fixed **three genuine WCAG AA contrast
  failures** — `text-gray-400` on the builder *Discard* button and the dashboard *what-if*
  label (≈2.9:1), and a `text-gray-300` separator in Compare (≈1.6:1) — bumped to
  `text-gray-600` / `text-gray-500` (+ `aria-hidden` on the decorative separator). The
  Pa11y/axe-4.2 "color-contrast" error on the public pages was confirmed a **false positive**
  (those pages carry no text below the AA threshold).

## Extending the automated coverage

The forecast result pages (`/scenarios/{id}/results`, `/compare`) and `/account/security`
need a concrete scenario id and (for the security page) a password confirmation; the
authenticated forecast pages are best checked with axe DevTools in the browser, since
scripted login under Pa11y CI's old Puppeteer is fragile and its axe is not current. The
rendered ApexCharts canvases always need a manual real-browser pass (only the accessible
tables/text are machine-checkable).

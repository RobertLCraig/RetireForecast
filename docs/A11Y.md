# Accessibility (WCAG 2.1 AA) checking

Accessibility is a hard constraint for this project: every result renders as headline
text + an accessible `<table>` alongside the charts, the layout carries a skip link and
landmark structure, and forms wire `aria-invalid` / `aria-describedby`. This doc covers
how to check it.

## The authoritative check: axe DevTools / Lighthouse in your browser

For the go-live verification pass, run **current axe in the Chrome you already have** — it
is trustworthy, needs no downloads, and covers the authenticated forecast pages trivially
(just log in first):

- **axe DevTools** (free Deque browser extension): open a page, DevTools → *axe DevTools* →
  *Scan all of my page*. Do this for the landing + auth screens, then log in (the seeded
  `demo@example.com` / `password`) and scan `/dashboard`, `/scenarios/create`, a results
  page, `/account/security`.
- **Lighthouse** (built into Chrome DevTools): DevTools → *Lighthouse* → tick *Accessibility*
  → *Analyze*.

Fix the markup the report points at, not the report.

## Automated sweep (Pa11y CI)

[`.pa11yci.json`](../.pa11yci.json) drives [Pa11y CI](https://github.com/pa11y/pa11y-ci)
(axe runner, WCAG2AA) over the public pages + `/welcome`, with `npm run a11y`, and
[`.github/workflows/a11y.yml`](../.github/workflows/a11y.yml) runs it on push/PR (dormant
until the repo has a GitHub remote).

**Known limitations — treat its output as leads, confirm in axe DevTools:**

- **It cannot run on this machine as-is.** npm here is configured with `ignore-scripts=true`,
  so package postinstall scripts are skipped and no headless browser binary (puppeteer's
  Chromium, chromedriver) is ever downloaded. CI on Linux runs scripts, so it works there.
  To run it locally anyway, point it at your installed Chrome:
  ```powershell
  $env:PUPPETEER_EXECUTABLE_PATH = "C:\Program Files\Google\Chrome\Application\chrome.exe"
  php artisan serve            # in one shell (seed demo first: see below)
  npm run a11y                 # in another
  ```
- **Its bundled axe-core is old (4.2, 2021)** and has emitted at least one **false positive**
  here (a contextless "color-contrast" error on the public pages, which have no sub-AA text).
  This is why axe DevTools (current axe) is the authoritative check, and Pa11y CI is only a
  coarse regression smoke.

Authed pages need the seeded demo account:
```powershell
php artisan db:seed --class="Database\Seeders\DemoScenarioSeeder"   # demo@example.com / password
```

## Sweep log

- **2026-06-28:** first local sweep. Found and fixed **three genuine WCAG AA contrast
  failures** — `text-gray-400` on the builder *Discard* button and the dashboard *what-if*
  label (≈2.9:1), and a `text-gray-300` separator in Compare (≈1.6:1) — bumped to
  `text-gray-600` / `text-gray-500` (+ `aria-hidden` on the decorative separator). The
  Pa11y/axe-4.2 "color-contrast" error on the public pages was confirmed a **false positive**
  (those pages carry no text below the AA threshold).

## Extending the automated coverage

The forecast result pages (`/scenarios/{id}/results`, `/compare`) and `/account/security`
need a concrete scenario id and (for the security page) a password confirmation; the
authenticated forecast pages are best checked with axe DevTools in the browser, since
scripted login under Pa11y CI's old Puppeteer is fragile and its axe is not current. The
rendered ApexCharts canvases always need a manual real-browser pass (only the accessible
tables/text are machine-checkable).

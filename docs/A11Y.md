# Accessibility (WCAG 2.1 AA) checking

Accessibility is a hard constraint for this project: every result renders as headline
text + an accessible `<table>` alongside the charts, the layout carries a skip link and
landmark structure, and forms wire `aria-invalid` / `aria-describedby`. This doc covers
the automated sweep that guards it.

## What runs

[Pa11y CI](https://github.com/pa11y/pa11y-ci) drives a headless Chrome over the rendered
pages with two engines, **axe-core** and **HTML CodeSniffer**, against the **WCAG2AA**
standard. The page list + login scripting live in [`.pa11yci.json`](../.pa11yci.json).

It covers the public pages with no setup (`/`, `/login`, `/register`, `/forgot-password`)
and the authed shell pages (`/welcome`, `/dashboard`, `/scenarios/create`) by scripting a
login + disclaimer-acknowledgement with the seeded demo account.

> **Status:** scaffolded but not yet run green in this environment (it needs a headless
> Chrome + the served app). Run it locally per below, or let the CI workflow run it once
> the repo has a GitHub remote.

## Run it locally

Pa11y needs the **built, served** app and the **demo account** (so the authed pages can
log in). From the project root:

```powershell
npm install                 # first time: pulls pa11y-ci (downloads a Chromium)
npm run build
php artisan migrate:fresh --seed
php artisan db:seed --class="Database\Seeders\DemoScenarioSeeder"   # creates demo@example.com / password
php artisan serve           # serves on http://127.0.0.1:8000
```

Then, in a second shell:

```powershell
npm run a11y
```

Each URL passes or reports its WCAG violations (rule, selector, context). Fix the markup,
not the report.

## CI

[`.github/workflows/a11y.yml`](../.github/workflows/a11y.yml) runs the same sweep on push /
PR. It is dormant until the project has a GitHub remote (it is local-first today).

## Extending coverage

The forecast result pages (`/scenarios/{id}/results`, `/compare`) and the account-security
2FA page need a concrete scenario id and, for `/account/security`, a fresh password
confirmation; add them to `.pa11yci.json` with the demo scenario's id and the extra
`set field #password` + `click element button[type=submit]` confirm step. The rendered
ApexCharts canvases still need a manual real-browser pass (only the accessible tables/text
are machine-checkable).
